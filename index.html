The Problem

Charity is broken by design

Traditional platforms suffer from opacity, high fees, and slow settlement — eroding donor trust and draining resources.

🕶

Opacity

Donors cannot verify how money is spent. Annual audits are the only transparency mechanism.

💸

High Fees

Payment processors, banks, and intermediaries take 3-8% of every donation.

⏳

Slow Settlement

Cross-border disbursements take 3-5 business days via SWIFT networks.

📋

Limited Accountability

No real-time tracking. No programmable spending controls. Just trust.

🔀

Complex Reconciliation

Multiple bank accounts, currencies, and reporting entities create overhead.

🌍

Limited Reach

SWIFT fees and correspondent banking make small cross-border transfers unviable.

The Solution

Stablecoins as the settlement layer

EUR stablecoins across multiple chains for treasury, disbursement, and spending — with every transaction publicly verifiable on-chain.

🔍

Radical Transparency

Every disbursement is an on-chain transaction with public proof. Real-time verifiability, not annual audits.

⚡

Near-Zero Fees

Base L2 at $0.005/tx, Solana at $0.00025/tx. No SWIFT fees, no correspondent banking chains.

⏱

Real-Time Settlement

2s on Base, 400ms on Solana. No T+2 banking delays for fund disbursements.

⚙

Programmable Rules

Spending controls enforced at wallet level — MCC restrictions, daily limits, approval workflows.

🌐

Global Reach

Stablecoins enable cross-border disbursement without SWIFT fees or correspondent banks.

🏦

Familiar Payments

Donors pay via card, bank transfer, or iDEAL. Organizers spend via Visa/MC card or IBAN withdrawal.

Architecture Pattern

Hexagonal architecture per service

Application Layer

Controllers, Kafka Consumers, Temporal Activities. Thin adapters that translate external requests into domain operations.

Domain Layer (Pure)

Entities, Value Objects, Domain Services, Ports. No framework dependencies. Business logic lives here.

Infrastructure Layer

JPA Repositories, REST API Clients, Blockchain Adapters, Kafka Producers. Implements domain ports.

Key Principles

  • Domain layer is pure — no framework coupling
  • Ports define interfaces in domain
  • Adapters implement ports in infrastructure
  • No cross-service database access
  • Services communicate via Kafka or REST

Architecture Decisions

Key ADRs

ADR-001
Stablecoin Selection
EURC (Circle) + EURe (Monerium)
Both MiCA-licensed. EURC: largest EUR stablecoin, native on Base/Arbitrum/Ethereum/Solana. EURe: IBAN-native for seamless SEPA.
ADR-002
Custody Solution
Fireblocks MPC-CMP
No single private key. SOC 2 Type II. Built-in policy engine. Wallet-as-a-Service for organizer wallets at scale.
ADR-003
Blockchain Network
Multi-chain: Base + Solana + ETH + Avalanche
Base primary (114 TPS, $0.005). Solana high-throughput (2,600 TPS, $0.00025). ETH L1 reserve. Avalanche expansion. All have native EURC.
ADR-004
Orchestration
Temporal (self-hosted)
Durable execution survives crashes. Built-in retries + saga compensation. Deterministic replay for full audit trail.
ADR-005
Payment Rails
Multi-provider strategy
Stripe (cards), Mollie (iDEAL), banking partner (SEPA). Port/adapter pattern makes provider swap trivial.
ADR-007
On/Off-Ramp
Circle Mint (batch) + Monerium
Circle: batch EUR wire → EURC mint (daily settlement). Monerium: IBAN-native EURe for smaller real-time flows. No per-donation minting.
ADR-008
Database Strategy
PostgreSQL per service
Full autonomy. Independent scaling. JSONB for flexibility. pg_crypto for column-level PII encryption.
ADR-009
Event Bus
Kafka (AWS MSK)
Durable ordered log with replay. Transactional Outbox for exactly-once. AWS-managed, no operational burden.
ADR-011
Transparency
Hybrid on-chain / off-chain
Financial flows on-chain (verifiable). PII off-chain (GDPR). Merkle root anchoring for off-chain integrity proof.

  • Outbox records: transactional event publishing
  • Ledger entries: double-entry bookkeeping
  • Compliance audit log: immutable decision history
  • On-chain transactions: immutable blockchain record
